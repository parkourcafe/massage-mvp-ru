# ТЗ / Архитектура — Massage Marketplace MVP

Profile-first продукт для **независимых профессиональных массажистов** и
клиентов. Массажист регистрируется, вставляет готовый текст из Avito,
Profi.ru, VK, Instagram, Telegram или резюме, AI извлекает структурированные
поля, массажист проверяет и подтверждает данные, публикует SEO-страницу и
делится ссылкой. Это **не** платёжный маркетплейс: оплаты сеансов и escrow
внутри нет — деньги клиент платит массажисту напрямую. В продукте есть
единый booking-флоу (переписка + согласование первого сеанса), но
**не** онлайн-оплата сеанса.

**Только профессиональный wellness / лечебный / спортивный / СПА массаж.**
Эротический, «адалт», сексуальный контент и «спец-услуги» запрещены и
блокируются автоматической модерацией + ручной проверкой. Это ключевое
отличие продукта от репетиторского эталона.

Доменный маппинг от репетиторского MVP:

| Репетиторы | Массаж |
|---|---|
| Tutor / репетитор | Therapist / массажист |
| Student / ученик | Client / клиент |
| Subject / предмет | Massage service / модальность массажа |
| Lesson / занятие | Session / сеанс |
| First meeting / первая встреча | Booking / первый сеанс |
| Inquiry / обращение | Booking inquiry / заявка на запись |
| Student CRM | Client CRM |
| Private lesson feedback | Приватные заметки по сеансу / отзыв клиента |

## Стек

- Next.js (App Router) + React + TypeScript + Tailwind
- Supabase Auth / PostgreSQL / Storage. Схема и RLS —
  `supabase/migrations/0001_init.sql`, `0002_rls.sql`
- OpenAI API (опционально), YooKassa, Vercel/Netlify
- Слой данных: канонический in-memory store с сидами
  (`src/lib/db/store.ts`, `src/lib/db/seed.ts`, фасад `src/lib/db/index.ts`),
  та же сигнатура, что у будущего Supabase-адаптера (`src/lib/supabase.ts`)
- AI: OpenAI при `OPENAI_API_KEY`, иначе детерминированный движок
  (`src/lib/ai.ts`) — продукт, включая извлечение из текста, работает офлайн

## Запуск

```bash
npm install
cp .env.example .env.local   # все переменные опциональны
npm run dev                  # http://localhost:3000
npm run typecheck && npm run test && npm run build
```

`.env.local` подключает Supabase, OpenAI и YooKassa для прода.

## Единый booking-флоу (запись = переписка + согласование первого сеанса)

Один поток, без отдельных модулей чата и календаря. Клиент создаёт заявку
(`/therapist/[slug]/booking`, публично, `noindex`), получает защищённую
токеном страницу `/booking/[token]` (`noindex`) с перепиской,
согласованием времени и отменой. Массажист ведёт всё в
`/dashboard/bookings` и `/dashboard/bookings/[id]` (auth, `noindex`,
доступ только владельцу профиля): отвечает, предлагает/принимает время,
подтверждает сеанс, отмечает исход и конвертирует в клиента
(создаётся запись `clients`).

Статусы (`BookingStatus`): `new, chat_started, waiting_therapist_reply,
waiting_client_reply, time_requested, time_proposed, confirmed, completed,
converted_to_repeat_client, lost, cancelled, no_show`.
Исходы (`BookingOutcome`): `completed_good, completed_repeat_expected,
completed_not_fit, cancelled_by_client, cancelled_by_therapist, no_show,
rescheduled, lost_no_reply`. Каждое изменение пишется в `booking_events`,
что показывает всю историю записи.

Точный адрес массажиста и клиента публично **не** раскрывается; адрес
клиента виден массажисту только после статуса `confirmed`. CTA
«Записаться и согласовать первый сеанс» — на профиле, `/match/results`
и `/favorites`. После создания: «Заявка создана. Здесь вы сможете
переписываться с массажистом и согласовать первый сеанс.»

Схема — `bookings`, `booking_messages`, `booking_events`.
**Статус: готово.** Доделать: при необходимости сблизить
3-шаговое согласование сеанса с эталоном (`time_requested → time_proposed
→ confirmed → completed`) и типизировать `booking_events` (сейчас строки).

## Формат сеансов и локация

Профиль: мульти-выбор форматов (выезд к клиенту / приём у массажиста /
салон-студия / онлайн-консультация), страна, город, район,
метро/ориентир, публичная подпись локации, выезд к клиенту и районы
выезда, часовой пояс. Точный приватный адрес хранится отдельно и
**никогда** не выводится публично — на профиле видны только город,
район, метро, публичная подпись и районы выезда. Каталог фильтрует по
формату и району/метро; AI-подбор учитывает формат и совместимость по
городу/району для выездного/очного (жёсткие фильтры) и удобство
выезда (мягкая оценка). Схема — поля локации в `profiles`.
**Статус: готово.**

## Биллинг и подписки (YooKassa)

Тарифы Free / Pro 490 ₽ / Expert 990 ₽ (`src/lib/plans.ts`). MVP —
разовая помесячная оплата; рекуррентность смоделирована, но **отключена**
до согласования с провайдером. Поток: `/dashboard/billing` → создаётся
платёж (pending) → YooKassa checkout → **вебхук**
`/api/payments/webhook/yookassa` → проверка платежа на стороне
провайдера (`verifyYooKassaPayment`) → активация подписки (30 дней),
`profile.plan`, инвойс. Pro **не** активируется по фронтовому редиректу
— только после верифицированного вебхука. Вебхук идемпотентен через
`payment_events`. Без ключей YooKassa — mock-режим
(`/dashboard/billing/mock-checkout` дергает вебхук, весь флоу проверяется
офлайн). Ключи только в env, на фронт не попадают.
`/api/subscriptions/cancel` снимает авто-продление (доступ до конца
периода). Просрочка → откат на Free. Гейты в `src/lib/plans.ts`
(`FREE_LIMITS`, доступ к SEO-индексации, заявкам, полному медиа,
CRM, аналитике, поддержке менеджера). Админ: `/admin/payments`,
`/admin/subscriptions`. Схема — `plans, subscriptions, payments,
payment_events, invoices`.
**Статус: готово.** Обязательный чекбокс оферты в `/dashboard/billing`
(кнопки оплаты заблокированы без согласия) с enforce на сервере в
`/api/payments/create` (400 без `offerAccepted`); публичные
индексируемые страницы `/terms`, `/privacy`, `/offer`,
`/subscription-terms` с перелинковкой.

## Медиа профиля — фото и видео

`/dashboard/media` («Фото и видео» в меню кабинета).

**Фото:** фото профиля, галерея, сертификаты/дипломы, скриншоты
отзывов, документы. Допустимо JPG/PNG/WebP до 5 МБ; изображения для
MVP сохраняются локально в `public/uploads` (каталог в репозитории,
файлы в `.gitignore`); замена на Supabase Storage — точечная. Лимиты
и предупреждение о правах на публикацию (в т.ч. фото клиентов и согласие
на съёмку) показаны в интерфейсе. На Free действует лимит
`FREE_LIMITS.maxMedia`.

**Видео:** добавление видео **по ссылке** (YouTube, VK Video, Rutube,
Vimeo) — без загрузки видеофайлов. Типы медиа `intro_video` /
`session_video`. Публичный профиль показывает вступительное видео
(embed), фото, галерею, сертификаты и скриншоты отзывов — только после
публикации профиля. Медиа учитываются в SEO-score.

Схема — `profile_media` (тип, url, подпись, порядок, публичность).
**Статус: частично.** Готово: типы медиа, видео-по-ссылке, embed на
профиле, гейт по публикации. Доделать: реальная загрузка изображений в
`public/uploads`, валидация JPG/PNG/WebP ≤ 5 МБ, предупреждение о правах/
согласии, enforce `FREE_LIMITS.maxMedia` в `addMedia`, embed для
VK/Rutube/Vimeo (сейчас полноценно только YouTube).

## Контакты и соцсети (WhatsApp, Telegram, VK, сайт)

В отличие от репетиторского эталона (где контакт только через
заявку), массажисту нужны публичные каналы связи. На профиле —
блок «Связаться»: WhatsApp (кнопка `wa.me/<номер>`), Telegram, VK,
Instagram, личный сайт + основной CTA на booking-форму. Массажист
указывает их в конструкторе профиля; пустые каналы не показываются.
Клики по каналам трекаются (`contact_clicks`, см. «Аналитика»).
Телефон/точный адрес остаются приватными до согласования сеанса.

Схема — колонки `whatsapp`, `telegram_url`, `vk_url`,
`instagram_url`, `website_url` в `profiles`; поля в `ProfileEditor`;
блок на `/therapist/[slug]`.
**Статус: нет — см. роадмап (приоритет).**

## Приватная взаимная обратная связь

**Никогда не публичная.** Публичных отзывов/рейтингов нет, ничего не
выводится в публичном профиле массажиста.

- Массажист: `/dashboard/clients` (список, создание),
  `/dashboard/clients/[id]` — заметки по сеансам и **структурированная
  приватная оценка клиента** (состояние/жалобы, переносимость нагрузки,
  динамика, рекомендации, следующий шаг, заметка). Видно **только
  массажисту**.
- Клиент: `/client/[token]` — без авторизации, по секретному токену
  (`crypto.randomBytes(24).base64url`, `unique` в схеме), страница
  `noindex` и закрыта в `robots.txt`. Поля: самочувствие после сеанса,
  комфорт, интенсивность (мало/норма/много), комментарий, пожелание,
  продолжать / пауза / обсудить.
- Видимость: оценка массажиста о клиенте — только массажисту; обратная
  связь клиента — массажисту и на его токен-странице. Доступ массажиста
  жёстко scoped по `profile_id` (нужен тест на границу владения).
- Админка ничего не публикует и по умолчанию не показывает эти данные.
- UX: метки «Приватная заметка», «Заметка по сеансу», «Динамика»;
  явный текст «Эти заметки не публикуются на сайте».

Схема — `therapist_private_session_notes`, `client_private_feedback`,
`clients.token` (`0001_init.sql`, RLS в `0002_rls.sql`,
`client_id`-связка в `0003_feedback_linkage.sql`).
**Статус: готово.** Страница `/client/[token]` (`ClientFeedbackForm`),
API `/api/client-feedback` (с модерацией комментария), структурированная
приватная оценка и просмотр входящих отзывов в `/dashboard/clients/[id]`
(`ClientDetail`), API `/api/dashboard/private-note`, store-функции
(`submitClientFeedback`, `addTherapistPrivateNote`,
`listClientFeedbackForProfile`, `listTherapistPrivateNotes`,
`getClientByToken`), выдача `clients.token` при конверсии заявки.
Тесты — `tests/feedback.test.ts` (включая границу владения).

## Импорт текста (ключевая функция)

`/dashboard/import`: вставка текста + выбор источника (Avito, Profi.ru,
VK, Instagram, Telegram, резюме) → `POST /api/import` `{task:"extract"}`
→ структурированные поля → форма проверки/правки → «Применить к профилю
(черновик)». Ничего не публикуется автоматически; статус остаётся
`draft` до явной публикации/модерации.
**Статус: готово.**

переписываются и **не** генерируются. Все вызовы логируются в
`ai_generations`.
**Статус: готово (без лога).** Доделать: таблица `ai_generations`
и запись вызовов из `src/lib/ai.ts`.

## Модерация (отличие от эталона — обязательна)

`src/lib/moderation.ts`, `/admin/moderation`. Эротический/«адалт»/

## v1 (post-MVP): AI-подбор массажиста

Аддитивный слой, не нужен для первого релиза. Клиент заполняет короткую
анкету (`/match`): зона/проблема (спина, шея, спорт-восстановление,
релакс, лимфодренаж, беременность и т.п.), цель, формат, город при
очном, бюджет, чувствительность к нажиму, пол массажиста (если важно).
Жёсткие фильтры → `src/lib/matching.ts` оценивает мягкое соответствие
→ до 3 массажистов с `match_score`, причинами и рисками
(`/match/results`).

Гарантии (зашиты в код): подбор не подаётся как гарантированный
(дисклеймер); у каждого результата есть причины **и** риски/вопросы;
без чувствительного медицинского профилирования — сравниваются только
явно указанные поля; ранжирование **не** использует платный тариф.

Схема — `match_requests`, `match_results`; видимость входящих подборов
в кабинете массажиста.
**Статус: частично.** Подбор считается; персистентности (`match_requests`/
`match_results`) и экрана входящих подборов нет.

## Авторизация

`/api/auth/{login,signup,logout}`, `users`-таблица
(`0004_auth.sql`). Тесты — `tests/auth.test.ts`.
**Статус: готово.**
